# moshdrop 设计文档

日期：2026-07-03　状态：已获用户批准（本文档为定稿存档）

## 1. 背景与问题

mosh 是屏幕状态同步器而非字节管道，任何依赖终端转义序列的带内文件传输协议（zmodem/lrzsz、trzsz、iTerm2 上传、kitty 传输协议）都无法穿过 mosh。这是 mosh 上游挂了多年的开放请求(2021/2022 起)（mobile-shell/mosh#1184、#1135），上游近乎休眠（上一版本 1.4.0 发布于 2022 年），等待官方协议内整合不现实。

同时，2025-2026 年大量用户在远程机器上通过 mosh + tmux 运行终端 AI Agent（Claude Code、Codex CLI 等），"把本地截图/文件喂给远端 Agent"成为高频新需求。现有唯一接近的方案 trzsz-ssh (tssh) 要求整体替换 mosh，对依赖 mosh 抗弱网能力的用户不可接受。

**结论：存在真实产品空白。定位一句话：Keep mosh. Just drop files.**

## 2. 目标与非目标

### 目标
- 用户把任意本地文件（截图浮窗、Finder 文件、多选文件）拖进正在跑 mosh 的终端窗口，远端收到的是**在远端真实可用的路径**。
- 对 mosh 使用体验零侵入：`moshdrop <mosh 参数原样>`，其余时间逐字节透明。
- 远端零安装、零配置：复用用户现有 ssh 配置与密钥。
- 可开源（MIT）：不写死任何主机名，单二进制分发。

### 非目标
- 不做协议内传输（不改 mosh）。
- 不做远端→本地方向的下载（v1 不做，留作后续）。
- 不做 Linux/Windows 本机端支持（v1 只保 macOS 本机端；远端任意平台）。
- 不做目录上传（v1 只传文件；目录原样放行）。

## 3. 用户体验

```
$ moshdrop ccc            # 用法与 mosh 完全一致（可 alias mosh=moshdrop）
# …正常使用 mosh/tmux/CC…
# 截图 → 把右下角浮窗直接拖进窗口 → 一两秒后输入框出现远端路径
# → Claude Code 显示 [Image #N]，可直接看图
```

- 拖 1 个或 N 个文件都支持，注入顺序与拖入顺序一致。
- 上传失败：原始粘贴照常放行（fail-open）+ macOS 系统通知说明原因。
- 体检：`moshdrop doctor <host>` 全链路自检（详见 §6）。

## 4. 架构

全部逻辑在本机，远端零安装：

```
终端(Ghostty/iTerm/任意) ── stdin 字节流 ──▶ moshdrop（Go 单二进制）
  ① pty 代理：spawn `mosh <原样参数>`，双向透传（窗口尺寸/信号/退出码）
  ② 粘贴探测：识别"整段粘贴 = 1..N 个本地真实存在的文件路径"
  ③ 上传器：经 ssh（ControlMaster 热连接）scp 到远端 drop 目录
  ④ 注入器：把换写后的远端路径按原粘贴语义写入 mosh 输入流
                                  ──▶ mosh-client ─UDP─▶ mosh-server ▶ tmux ▶ CC/shell
旁路：ssh <与 mosh 相同的目标>（复用 ~/.ssh/config，无新认证）
```

### 关键决策
1. 用法与 mosh 一致，参数原样透传。（工作名 moshdrop，**最终名称待定**）
2. ssh 目标自动推导自 mosh 目标（`[user@]host`、`--ssh=` 均支持）——用户敲 `moshdrop ccc`，上传就走 `ssh ccc`，同名同配置，零额外设置。
3. 远端落点默认 `~/.moshdrop/`（首次自动 mkdir，**目录名待定**），可按 host 覆盖。不依赖两端路径一致。
4. fail-open + 自动诊断（§6）。
5. Go 单二进制，brew/直接下载分发。
6. 部署后可拆除用户现有的同步文件夹 + launchd + 截图改位置等临时方案。

## 5. 粘贴探测规则（核心）

拦截当且仅当**全部**满足：
1. 内容是一段完整粘贴（括号粘贴 `ESC[200~…ESC[201~` 为主通道；非括号粘贴用"单次读入整块"启发式做后备，走同样验真）；
2. 还原终端转义（如 `\ ` 空格）后，整段内容恰好切分为 1..N 个绝对路径；
3. 每个路径经本地 `stat()` 验证为真实存在的普通文件。

任何一条不满足 → 逐字节原样放行。用户想把本地路径当纯文本发送：前后加任意字符（如引号）即可绕过。误伤防线是"真的去硬盘看一眼"，误判率≈0。

## 6. 错误处理与自动诊断

| 层 | 机制 |
|---|---|
| 运行时 | fail-open：失败即原样放行原始粘贴 + macOS 通知（含原因与下一步建议）。绝不吞输入、绝不卡终端。 |
| 事件日志 | 每次拖放记 JSONL 一条（时间/主机/文件/字节/耗时/结果/错误类别）→ `~/.moshdrop/events.log`。 |
| doctor | `moshdrop doctor <host>`：① mosh 二进制/版本 ② 推导出的 ssh 目标 ③ 免密连通+延迟 ④ 远端 drop 目录可写 ⑤ 传 1KB 探针文件并校验哈希（端到端真链路）⑥ ControlMaster 可用性 ⑦ 解读最近一次失败并给修复建议。逐项 PASS/FAIL。诊断代码与运行时共用同一链路。 |

## 7. 上传与注入细节

- 启动时建立 ssh ControlMaster 长连接；每次拖放走热连接（小文件延迟目标 <1s）。
- 远端文件名=原文件名（保留扩展名，CC 依赖它识别图片）；重名加短后缀；含换行符的文件名拒绝并通知。
- 多文件并发上传，全部完成后一次性注入；保持括号粘贴语义（CC 侧与本地拖拽行为一致）。
- 文件名处理走 API 不拼 shell 字符串，杜绝注入。

## 8. 安全边界

- 仅"整段粘贴=本地真实文件"触发；手敲文本/粘贴文字永不触碰。
- 仅上传用户亲手拖入的文件；远端只写配置的 drop 目录。
- 无后台常驻：随 mosh 会话启动/退出，不留进程。

## 9. 测试计划

- **单元**：探测器对抗样例（形似路径的文本、转义边角、多文件、ESC 序列跨读切碎）；mosh 参数→ssh 目标推导表（user@host/别名/--ssh=/IPv6）。
- **集成**：pty 测试床双向灌随机字节验证逐字节透明；ssh localhost 端到端拦截-上传-注入。
- **人肉矩阵**：Ghostty/iTerm2/Terminal × CC/zsh/vim × 截图浮窗/多选/中文带空格文件名/大文件/传输中断网。
- **实战**：用户在 ccc 环境日用一周 dogfood。
